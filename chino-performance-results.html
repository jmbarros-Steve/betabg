Qué había antes

El archivo performance.ts tenía una sola función que hacía GET a una URL y medía el tiempo. Si el check no tenía url y max_ms en su config, retornaba "skip".

Solo 2 checks funcionaban:

Los otros 12 checks del Grupo A tenían check_config: {} (vacío). No tenían URL ni tiempo máximo. El código decía "no sé qué medir" y los saltaba:

// ANTES — lo unico que sabia hacer: if (!url || !maxMs) return { result: 'skip' }; // ← los 12 checks caian aca const res = await fetch(url); // ← solo GET, sin body, sin auth

Qué se hizo

Se reescribió performance.ts para que cada check tenga su propia lógica. Ya no depende de que el check tenga config — el código sabe qué endpoint testear, cómo llamarlo, y cuánto debe tardar como máximo.

ANTES (62 líneas)
// 1 sola funcion // Solo sabe hacer GET a una URL // Si no tiene config → skip // Sin auth, sin POST, sin body executePerformance(check) { url = check.check_config?.url // vacío maxMs = check.check_config?.max_ms // vacío if (!url || !maxMs) → skip ❌ }
AHORA (246 líneas)
// switch por check_number // Cada check sabe su endpoint, method, body // Auth con service_role_key // POST real a cada servicio executePerformance(check) { switch (check.check_number) { case 60: → POST /api/klaviyo-push-emails case 66: → POST /api/send-email case 71: → POST /api/steve-chat + verificar respuesta case 181: → POST /api/steve-chat case 182: → POST /api/fetch-shopify-products ...12 checks más... case 195: → POST api.anthropic.com (directo) } }

Cómo funciona cada check

Todos siguen el mismo patrón simple:

  1. El Chino hace POST al endpoint de Steve con autorización
  2. Prende el cronómetro
  3. Espera la respuesta
  4. Para el cronómetro
  5. Si tardó menos del máximo → PASS
  6. Si tardó más del máximo → FAIL "Tardó Xms, máximo Yms"
  7. Si el servidor respondió 500 → ERROR "Servidor roto"

Detalle técnico (simple)

❓ ¿Por qué POST con body vacío en algunos?

Endpoints como "enviar email" o "generar imagen" no se pueden ejecutar de verdad en una prueba (enviaría un email real o gastaría créditos). Entonces El Chino manda un POST vacío — el servidor dice "te faltan datos" (error 400) pero responde rápido. Si está caído o sobrecargado, ni siquiera podría responder eso.

❓ ¿Por qué #71 sí manda mensaje real?

Porque la descripción dice "responde con mensaje válido". Steve Chat es seguro de llamar — solo genera texto, no tiene efectos secundarios. El Chino le manda "ping" y verifica que la respuesta no esté vacía.

❓ ¿Por qué #195 llama directo a Anthropic?

Porque ese check mide la latencia de Claude, no de Steve. Llama a la API de Anthropic con el modelo más barato (Haiku), le dice "hi" y pide máximo 5 tokens. Cuesta menos de $0.001 por prueba.

❓ ¿Qué pasa si un endpoint responde 400?

Se cuenta como PASS (si fue rápido). El check mide velocidad, no si los datos son correctos. Un 400 rápido = servidor vivo y responsivo. Solo un 500 o timeout cuenta como error.

❓ ¿Por qué 3 ya pasaron y 9 dicen "próximo ciclo"?

El Chino ejecuta los checks en lotes. Los checks #60, #66, #71 tienen plataformas distintas (klaviyo, stevemail, steve_chat) y corrieron en este patrol. Los #181-195 tienen plataforma "infra" y su lote corrió antes del deploy (a las 17:03). En el próximo ciclo (cada 30 min) ya corren con el código nuevo.
