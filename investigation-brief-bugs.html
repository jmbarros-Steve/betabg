BUG-01 Pregunta duplicada al avanzar Backend Critico ▶
Sintoma

Cuando Steve avanza de una pregunta a la siguiente, el usuario ve la misma pregunta DOS veces: una al final del mensaje de Steve (texto parcial) y otra completa debajo.

Root Cause

La logica de implicitAdvanceDetected (safety net agregado el 13/04) trunca el texto del AI cuando detecta que escribio la siguiente pregunta sin [AVANZAR], pero el truncado no siempre limpia todo el texto. Luego, el append deterministico agrega la pregunta completa de nuevo.

AI genera: "Excelente! [eval] Pregunta 3: Como defines..." | implicitAdvance detecta "Pregunta 3: Como defines" | Trunca en indexOf("Como defines") → queda "Excelente! [eval] Pregunta 3: " | hasAdvanced = true → append deterministico agrega: "-- Intro Steve -- Pregunta 3: Como defines tu audiencia?" | RESULTADO: "Excelente! [eval] Pregunta 3: \n-- Intro Steve -- Pregunta 3: Como defines..." ↑ DUPLICADO
Archivos Involucrados

cloud-run-api/src/routes/ai/steve-chat.ts:2070-2093

// Safety net: lineas 2070-2090
let implicitAdvanceDetected = false;
if (!assistantMessage.includes('[AVANZAR]') && !isRejection && !isLastQuestion) {
  const nextBriefQ = BRAND_BRIEF_QUESTIONS[currentQuestionIndex + 1];
  if (nextBriefQ) {
    const nextQSignature = nextBriefQ.question.substring(0, 50);
    const nextIntroSignature = (nextBriefQ.steveIntro || '').substring(0, 30);
    if (assistantMessage.includes(nextQSignature) || ...) {
      implicitAdvanceDetected = true;
      // PROBLEMA: trunca pero puede dejar texto parcial
      const truncIdx = introIdx !== -1 ? introIdx : sigIdx;
      if (truncIdx > 0) {
        assistantMessage = assistantMessage.substring(0, truncIdx).trim();
      }
    }
  }
}
Fix Propuesto

Opcion A: Truncar agresivamente

Buscar CUALQUIER fragmento de la siguiente pregunta (no solo los primeros 50 chars) y truncar ANTES de cualquier coincidencia. Tambien buscar el steveIntro y keywords como "Pregunta X:".

// Buscar multiples signatures para truncar mas agresivamente
const patterns = [
  nextBriefQ.question.substring(0, 50),
  nextBriefQ.question.substring(0, 30),
  (nextBriefQ.steveIntro || '').substring(0, 30),
  `Pregunta ${currentQuestionIndex + 2}`,  // "Pregunta 3:"
  `pregunta ${currentQuestionIndex + 2}`,
];
const truncIdx = Math.min(...patterns
  .map(p => p ? assistantMessage.indexOf(p) : -1)
  .filter(i => i > 0));
if (truncIdx < Infinity) {
  assistantMessage = assistantMessage.substring(0, truncIdx).trim();
}

Opcion B: Forzar [AVANZAR] al truncar

Si se detecta implicitAdvance, insertar [AVANZAR] al final del texto truncado. Asi el flujo normal de append funciona sin ambiguedad.

BUG-03 Secciones del brief sin analisis real Backend API Critico ▶
Sintoma

Varias secciones del brief muestran texto generico/placeholder en vez de analisis basado en datos reales de la marca. Secciones afectadas: propuesta de valor, estrategia creativa de competidores, keywords, SEO.

Root Cause: Falla silenciosa en scraping (Apify)

El endpoint analyze-brand-research usa Apify para scrapear el sitio web del cliente y competidores. Cuando Apify falla (timeout, sitio protegido, JS-heavy), el error se traga silenciosamente y el pipeline continua con data vacia. Claude entonces genera texto placeholder porque no tiene datos reales.

Cadena de Falla
1. analyze-brand-research llama Apify para scrapear URLs 2. Apify timeout / blocked / JS-render fail 3. El codigo NO verifica resp.ok ni lanza error 4. research_data queda vacio: { website: null, competitors: [] } 5. analyze-brand-strategy recibe data vacia 6. Claude genera placeholder: "La marca se posiciona como..." 7. BrandBriefView renderiza texto sin datos → parece vacio/generico

cloud-run-api/src/routes/ai/analyze-brand-research.ts — no verifica resp.ok

Problemas Especificos
  • No hay verificacion resp.ok despues del fetch a Apify
  • No hay fallback a Firecrawl cuando Apify falla (FIRECRAWL_API_KEY existe en env pero no se usa)
  • Sitios con Cloudflare/JS-heavy bloquean Apify consistentemente
  • No hay logging del contenido scrapeado → imposible debuggear sin reproducir
  • Cada falla de scraping cascadea a TODAS las secciones que dependen de datos web
Secciones Afectadas por Scraping Vacio
  • SEO y Keywords — sin datos del sitio, no puede analizar keywords
  • Propuesta de Valor — sin contenido real, genera texto generico
  • Competidores — sin scraping de competidores, no hay estrategia creativa
  • Tono de Voz — sin texto real del sitio, no puede inferir tono
Fix Propuesto

Implementar Firecrawl como fallback

// En analyze-brand-research.ts
async function scrapeUrl(url: string): Promise<string> {
  // Intento 1: Apify
  try {
    const apifyResult = await scrapeWithApify(url);
    if (apifyResult && apifyResult.length > 200) return apifyResult;
  } catch (e) { console.log('[scrape] Apify failed:', e.message); }

  // Intento 2: Firecrawl (fallback)
  try {
    const fcResult = await scrapeWithFirecrawl(url);
    if (fcResult && fcResult.length > 200) return fcResult;
  } catch (e) { console.log('[scrape] Firecrawl failed:', e.message); }

  // Intento 3: fetch basico (ultimo recurso)
  const resp = await fetch(url, { headers: { 'User-Agent': '...' } });
  return await resp.text();
}

  • El campo tone_of_voice del API viene como string narrativo, no como objeto estructurado

Crear normalizeBrandIdentity()

Funcion que parsee el texto narrativo de tone_of_voice y lo estructure en: tono principal, adjetivos, do's, don'ts, ejemplos.

BUG-05 SEO: scraping falla silenciosamente Backend API Alta ▶
Sintoma

La seccion SEO del brief aparece vacia o con datos genericos. El analisis SEO requiere datos del sitio web real.

Root Cause

Mismo root cause que BUG-03 (scraping Apify). El analisis SEO depende 100% de tener contenido real scrapeado del sitio web del cliente. Cuando Apify falla, no hay datos para analizar meta tags, estructura H1-H6, contenido, etc.

Impacto
  • Sin scraping → sin meta tags, titles, headers, contenido
  • Claude genera recomendaciones SEO genericas sin datos reales
  • Keywords sin fundamento en contenido existente
  • Oportunidades SEO inventadas en vez de basadas en gaps reales
Fix

Resuelto con BUG-03 (fallback Firecrawl). Adicionalmente, el scraper deberia extraer especificamente meta tags, H1-H6, y contenido principal para alimentar la seccion SEO.

BUG-06 Keywords incompletos en el PDF Frontend Alta ▶
Sintoma

La seccion de keywords en el PDF exportado sale incompleta o vacia, aunque en la vista web se muestran parcialmente.

Root Cause

La funcion renderKeywordPhases() referenciada en briefPdfSections.ts no existe o no esta implementada correctamente. El PDF usa un pipeline diferente al HTML para renderizar secciones.

Archivos

src/components/client-portal/BrandBriefView.tsx — vista HTML de keywords (parcial)

src/components/client-portal/briefPdfSections.ts — pipeline PDF (renderKeywordPhases missing)

Detalle
  • briefPdfSections.ts tiene las funciones que generan HTML para cada seccion del PDF
  • La seccion keywords requiere renderKeywordPhases() para fases de keywords (awareness, consideration, conversion)
  • Esta funcion falta o no maneja el formato de datos que entrega el backend
  • La vista web renderiza keywords de forma diferente (JSX directo) y no pasa por la misma logica
Fix Propuesto

Implementar renderKeywordPhases() en briefPdfSections.ts

Crear la funcion que transforme los datos de keywords normalizados en HTML compatible con html2canvas/jsPDF. Debe manejar las 3 fases (awareness, consideration, conversion) con sus respectivas keywords.

Root Cause

BrandBriefView.tsx renderiza el mapa perceptual como texto porque no hay componente SVG/chart implementado. El backend devuelve coordenadas (x, y) para cada marca en los ejes del mapa, pero el frontend solo los muestra como lista de texto.
