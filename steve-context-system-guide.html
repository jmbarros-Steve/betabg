5 Anatomía de un Context File

Qué secciones tiene cada agents/contexts/{agente}.md

Ejemplo: agents/contexts/felipe-w2.md

## Tus Tablas
Las tablas que este agente escribe. Con columnas clave, filas conocidas, estado actual. Solo las suyas (Felipe ve 8, no las 97).
## Tablas que Lees
Tablas de otros agentes que necesita consultar (read-only). Dice quién es el dueño y para qué la lee.
## Tus Crons
Crons de Cloud Scheduler que le pertenecen. Con schedule, endpoint, y estado conocido.
## Tus Archivos
Archivos de código que mantiene: backend, frontend, edge functions, libs. Con rutas exactas.
## Dependencias
De quién depende (agentes que le alimentan data) y a quién alimenta. El grafo de dependencias del agente.
## Problemas Conocidos
Tablas vacías, crons rotos, features desconectadas, env vars faltantes. Lo que hay que arreglar.
_shared.md tiene las convenciones que TODOS comparten: naming, imports, auth patterns, cross-review, archivos prohibidos, deploy commands. Cada context file solo tiene lo específico del agente.

7 Validación Automática (cada 12h)

El sistema se auto-mantiene. No tienes que hacer nada manualmente.

Context Validator

0 6,18 * * * (6am y 6pm UTC)

Compara tablas reales de Supabase y crons de Cloud Scheduler contra los context files.

Si encuentra algo nuevo sin dueño:
1. Lo agrega a _unassigned.md
2. Crea una task en Supabase con severity "medium"
3. El próximo agente que se active lo ve en Paso 5

_unassigned.md

Se llena automáticamente

Archivo que acumula tablas y crons nuevos que ningún agente tiene en su context.

Formato de cada entrada:
- [2026-04-04] tabla campaign_ab_tests — sin dueño
Cuando un agente se activa, revisa este archivo y decide si algo es suyo.

Qué detecta

Chequeos automáticos

Tres tipos de discrepancia:

1. Tabla sin dueño: existe en Supabase pero en ningún context
2. Tabla fantasma: está en un context pero no existe en Supabase
3. Cron huérfano: activo en Cloud Scheduler pero sin mención

Dónde vive

Dos versiones

Hay dos implementaciones para cobertura total:

Cloud Run: cloud-run-api/src/routes/cron/validate-contexts.ts
→ Lo llama el cron context-validator-12h

Local: scripts/validate-context-files.sh
→ Lo corres manualmente cuando quieras
8 Qué pasó con CLAUDE.md

De 690 líneas a 277. Qué se movió y qué se quedó.

Se movió a context files

  • Catálogo detallado de 97 tablas con columnas (~260 líneas)
  • Lista detallada de 44 crons con schedules (~90 líneas)
  • Historial de commits
  • Detalles de edge functions por agente

Se quedó en CLAUDE.md

  • Stack y deploy commands
  • Reglas de código y naming
  • Imports y auth patterns
  • Estructura del proyecto
  • Organigrama del equipo (nombres + roles)
  • Tabla resumen de 97 tablas (1 línea c/u, con dueño)
  • Cross-review obligatorio
  • SYNC agresivo a Supabase
  • Protocolo de activación (actualizado a 4 capas)
  • Env vars (20 obligatorias + 8 pendientes)
  • Archivos grandes (>30KB)
  • Delegación dinámica de sub-agentes
  • Protocolo de desafío
  • Bug → Task automático
CLAUDE.md sigue siendo la "biblia general". Todos los agentes lo leen siempre. La diferencia es que ahora tiene las reglas, no los datos. Los datos están en cada context file.

10 Preguntas Frecuentes
¿Qué pasa si creo una tabla nueva en Supabase? ▶
En máximo 12 horas, el cron context-validator-12h la detecta y la agrega a agents/state/_unassigned.md. El próximo agente que se active la verá en el Paso 5 de activación y podrá reclamarla. También crea una task en Supabase para que no se olvide.
¿Qué pasa si borro una tabla de Supabase? ▶
El validador la marca como "tabla fantasma" — existe en el context file pero no en Supabase. El script local (validate-context-files.sh) la puede auto-remover del context file. El endpoint Cloud Run solo reporta el warning.
¿Tengo que editar los context files manualmente? ▶
Casi nunca. Los context files son referencia estática — cambian cuando agregas tablas, crons, o archivos nuevos. El validador automático te avisa cuando hay discrepancias. Si un agente necesita actualizar su context, lo hace como parte de su trabajo normal. Tú solo intervienes para asignar items de _unassigned.md a un agente.
¿Qué es _shared.md? ▶
Son 84 líneas con convenciones que TODOS los agentes necesitan: naming conventions, import aliases, auth patterns, cross-review protocol, archivos prohibidos, deploy commands, SYNC protocol. Es como el "reglamento interno" que aplica a todo el equipo. Cada agente lo lee junto con su context personal.
¿Puedo ver todo esto en el admin? ▶
Sí. AdminOrganigrama fue actualizado con:
• Protocolo de Activación (4 capas) visual
• Sección "Automatizaciones" mostrando los 5 procesos automáticos
• Cada agente muestra cuántas tablas y crons tiene
• Tab "Contexto" en el detalle de cada agente con su context file y stats
¿Qué ganamos con esto? ▶
3 cosas concretas:

1. Menos ruido: Felipe (Meta Ads) ya no ve las 17 tablas de email. Ve sus 8 tablas y punto.
2. Auto-mantenimiento: Si alguien crea una tabla nueva, el sistema la detecta solo. No se pierde nada.
3. CLAUDE.md legible: 277 líneas vs 690. Las reglas generales claras, sin data dump.
¿Y si quiero agregar un agente nuevo (W20, W21...)? ▶
Tres archivos:
1. agents/personalities/nuevo-wXX.md — personalidad y misiones
2. agents/contexts/nuevo-wXX.md — tablas, crons, archivos (seguir el template de los existentes)
3. agents/state/nuevo-wXX.md — estado inicial vacío

Además, agregar sus tablas al mapa ASSIGNED_TABLES en cloud-run-api/src/routes/cron/validate-contexts.ts para que el validador lo reconozca.
