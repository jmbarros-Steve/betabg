Problemas de Seguridad 13
CRÍTICO Secreto de admin hardcodeado en código fuente Secrets ▼
cloud-run-api/src/routes/auth/admin-create-client.ts:18
El endpoint valida contra un secreto hardcodeado 'setup-jardin-eva-2026'. Este secreto es visible en el código fuente y quedará en el historial de git.
Fix: Usar variable de entorno ADMIN_SETUP_SECRET. Considerar protección adicional como IP whitelist.
CRÍTICO CORS permite todos los orígenes (origin: '*') CORS ▼
cloud-run-api/src/middleware/cors.ts:3-4
CORS configurado con origin: '*'. Esto habilita ataques CSRF y fuga de credenciales si se usan cookies o headers de autenticación.
Fix: Restringir a dominios específicos: origin: ['https://tudominio.com', 'https://app.tudominio.com']
CRÍTICO Archivo .env rastreado en historial de git Secrets ▼
.env (raíz del proyecto)
El archivo .env aparece como modificado en git status, lo que indica que fue commiteado previamente. Contiene claves de Supabase y URLs de API.
Fix: Revisar historial git con git log --all -- .env. Usar BFG Repo Cleaner para eliminar del historial. Rotar claves expuestas.
CRÍTICO Sin rate limiting en endpoints públicos DoS/Brute Force ▼
cloud-run-api/src/routes/auth/self-signup.ts, track-events.ts, email-unsubscribe
Endpoints públicos como signup, tracking de emails y unsubscribe no tienen rate limiting. Permite ataques de fuerza bruta, enumeración de emails y DoS.
Fix: Implementar middleware de rate limiting con sliding window. Usar Redis o in-memory store con límites por IP.
ALTO Open Redirect en tracking de clicks de email Redirect ▼
cloud-run-api/src/routes/email/track-events.ts:63-108
El endpoint de track click usa decodeURIComponent(targetUrl) y redirige sin validación. Un atacante puede crear links que redirijan a dominios maliciosos.
Fix: Validar que la URL destino pertenece a dominios permitidos usando new URL(decodedUrl).hostname contra whitelist.
ALTO JWT de Shopify sin verificación de firma Auth ▼
cloud-run-api/src/routes/shopify/sync-shopify-metrics.ts:54-60
Los session tokens de Shopify se decodifican con JSON.parse(atob(...)) sin verificar la firma JWT. Un atacante podría forjar tokens válidos.
Fix: Usar jsonwebtoken.verify() con la clave pública de Shopify para verificar la firma.
ALTO dangerouslySetInnerHTML sin sanitización XSS ▼
src/components/client-portal/email/ (7 ocurrencias en EmailTemplateBuilder, EmailBlockEditor, TemplatesPanel)
Se renderiza HTML de bloques de email usando dangerouslySetInnerHTML sin sanitizar el contenido. Si el HTML viene de fuentes no confiables, hay riesgo de XSS.
Fix: Usar DOMPurify para sanitizar todo HTML antes de renderizarlo: dangerouslySetInnerHTML={{__html: DOMPurify.sanitize(html)}}
ALTO OAuth state sin validación de timestamp CSRF ▼
cloud-run-api/src/routes/shopify/shopify-oauth-callback.ts:62-80
El nonce del estado OAuth se valida contra la DB pero sin verificación de timestamp. Permite ataques de replay con nonces antiguos.
Fix: Agregar validación de expiración: rechazar nonces con más de 10 minutos de antigüedad.
ALTO Procesador HTML custom sin librería de sanitización XSS ▼
cloud-run-api/src/lib/email-html-processor.ts
Implementación custom de parsing HTML con regex y operaciones de string (287+ operaciones). El parsing con regex es propenso a errores y puede no cubrir todos los vectores de ataque.
Fix: Reemplazar con librería probada como sanitize-html o DOMPurify.
MEDIO SSRF potencial en scraping de competidores SSRF ▼
cloud-run-api/src/routes/ai/analyze-brand-research.ts:236-242
URLs de competidores no se validan antes del scraping. Un atacante podría inyectar URLs internas (127.0.0.1, 10.*, 172.16.*) para atacar servicios internos.
Fix: Validar que los hostnames sean DNS públicos, rechazar IPs privadas y localhost.
MEDIO Errores internos expuestos al cliente Info Disclosure ▼
cloud-run-api/src/routes/oauth/meta-oauth-callback.ts:171
Las respuestas de error incluyen details: errorMessage que puede filtrar stack traces e información interna al cliente.
Fix: Solo retornar detalles en modo development: details: process.env.NODE_ENV === 'development' ? msg : undefined
MEDIO localStorage usado para datos sensibles sin encriptación Storage ▼
Múltiples componentes frontend (35 instancias)
Datos de usuario almacenados en localStorage/sessionStorage sin cifrado. Si hay XSS, toda la información es accesible.
Fix: Revisar qué datos se almacenan. Usar cookies HTTP-only para tokens. Considerar cifrado para datos sensibles.
MEDIO Variables de entorno no validadas al inicio Config ▼
cloud-run-api/src/index.ts, middleware/auth.ts
Se usa process.env.KEY! con non-null assertion sin validar al arrancar el server. Si falta una variable, falla en runtime.
Fix: Agregar función de validación al inicio que verifique todas las env vars requeridas.
Problemas de TypeScript 6
ALTO 1,257 usos de "any" en el codebase Type Safety ▼
Frontend: 137 "as any" en 31 archivos. Backend: extenso uso en routes.
El tipo any anula completamente el type-checking de TypeScript. Los archivos más afectados son BrandBriefView.tsx (68 ocurrencias), BlockConfigPanel.tsx (20), y múltiples archivos del backend.
Fix: Reemplazar any con tipos específicos o unknown con type guards. Crear interfaces para todas las respuestas de API.
ALTO TypeScript strict desactivado en frontend Config ▼
tsconfig.app.json
Frontend tiene strict: false, noImplicitAny: false, strictNullChecks: false. El backend sí tiene strict: true. Inconsistencia que permite bugs silenciosos en frontend.
Fix: Habilitar gradualmente: primero noImplicitAny, luego strictNullChecks, finalmente strict: true.
MEDIO Record<string, any> en 17 ubicaciones Type Safety ▼
CampaignCreateWizard.tsx, BrandBriefView.tsx, EmailTemplateGallery.tsx, BlockConfigPanel.tsx
Uso de Record<string, any> es un tipo demasiado permisivo que no ofrece autocompletado ni validación.
Fix: Definir interfaces específicas para cada caso de uso.
MEDIO Parámetros de función sin tipo (50+ ocurrencias) Type Safety ▼
BrandAssetUploader.tsx:177, ClientPortalConnections.tsx:40, EmailBlockEditor.tsx:87, y más
Funciones con parámetros tipados como (p: any) o sin tipo explícito. Elimina las ventajas de TypeScript.
Fix: Agregar tipos explícitos a todos los parámetros de función.
MEDIO useState sin tipo genérico (any[]) Type Safety ▼
29 ocurrencias: BrandBriefView, ShopifyDashboard, EmailTemplateBuilder, MetaAdCreator, MassCampaignsWizard
Hooks useState<any[]>([]) pierden toda la información de tipo del estado.
Fix: Crear interfaces para el estado: useState<Template[]>([])
MEDIO Magic strings sin enums ni constantes Type Safety ▼
Backend: manage-meta-campaign.ts, flow-engine.ts, y otros
Strings como 'create', 'pause', 'draft', 'subscribed' usados directamente sin constantes. Propenso a typos.
Fix: Definir enums: enum CampaignStatus { DRAFT = 'draft', ACTIVE = 'active' }
Anti-patrones React 7
ALTO 73+ inline function props causando re-renders Performance ▼
MetaAdCreator.tsx, CampaignCreateWizard.tsx, GoogleAdsGenerator.tsx, FinancialConfigPanel.tsx, MetaAudienceManager.tsx
Funciones inline en props onClick={() => {...}} crean nuevas referencias en cada render, causando re-renders innecesarios en componentes hijos.
Fix: Usar useCallback para handlers que se pasan como props a componentes hijos.
ALTO Dependencias faltantes en useEffect (115+ archivos) Hooks ▼
BrandAssetUploader.tsx:281, MetaAdCreator.tsx, OnboardingWizard.tsx:81-94, ClientPortalConnections.tsx:87
Múltiples useEffect con arrays de dependencias incompletos. Puede causar bugs sutiles donde los efectos no se re-ejecutan cuando deberían.
Fix: Agregar todas las dependencias necesarias al array. Usar el plugin de ESLint react-hooks/exhaustive-deps.
ALTO Doble fuente de verdad para estado (ref + state) State Management ▼
BrandAssetUploader.tsx:129-131
Usa analyzingRef y analyzing state simultáneamente para el mismo dato. Crea inconsistencias cuando uno se actualiza sin el otro.
Fix: Elegir una sola fuente de verdad. Usar ref si no necesita trigger de re-render, state si sí.
MEDIO Exceso de useState en componentes (10+ states) State Management ▼
ClientPortalMetrics.tsx, BrandBriefView.tsx, MetaAdCreator.tsx
Componentes con 10+ llamadas a useState. Dificulta el mantenimiento y puede causar renders en cascada.
Fix: Agrupar estados relacionados con useReducer o un objeto de estado.
MEDIO Props drilling a través de 4+ niveles Architecture ▼
CopyGenerator.tsx (clientId), CampaignStudio.tsx (estado)
clientId y otras props pasan por múltiples niveles de componentes sin necesidad.
Fix: Usar React Context para datos que necesitan muchos componentes, o composición de componentes.
MEDIO Sin Error Boundaries en árboles complejos Error Handling ▼
ClientPortalMetrics.tsx, CampaignStudio.tsx, y otros
Solo BrandBriefView.tsx tiene ErrorBoundary. Si un componente hijo falla en render, toda la app se cae.
Fix: Agregar Error Boundaries en los límites de cada feature principal del portal.
BAJO Componentes de función creados dentro del render Performance ▼
CopyGenerator.tsx:37-82 (renderBriefField)
renderBriefField es un componente funcional creado dentro del render. Se recrea en cada render perdiendo estado interno.
Fix: Extraer como componente separado fuera de la función padre.
Problemas de Backend 8
ALTO Un endpoint maneja 6+ acciones diferentes API Design ▼
cloud-run-api/src/routes/auth/admin-create-client.ts:10-261
POST /api/admin-create-client maneja setup_patricio, cleanup_patricio, fix_patricio, diagnostic, set_shop_domain, reset_password — todo en un solo endpoint de 261 líneas.
Fix: Separar en endpoints individuales: POST /api/admin/setup, POST /api/admin/cleanup, etc.
ALTO Sin validación de esquema en request bodies Validation ▼
Todas las rutas del backend
No hay validación de esquema en los bodies de las peticiones. Los campos requeridos no se verifican antes de usar, lo que puede causar errores en runtime.
Fix: Implementar validación con zod: const schema = z.object({ email: z.string().email() }); schema.parse(body);
MEDIO Formato de respuesta de error inconsistente API Design ▼
Todo el backend
Algunas rutas retornan { error: string }, otras { error: { message } }, y algunas incluyen details. Dificulta el manejo de errores en el frontend.
Fix: Estandarizar: { success: boolean, data?: T, error?: { code: string, message: string } }
MEDIO Lógica de HMAC duplicada en 3 archivos Duplicación ▼
middleware/shopify-hmac.ts, routes/email/flow-webhooks.ts, routes/shopify/shopify-oauth-callback.ts
La verificación de HMAC de Shopify está implementada en 3 lugares diferentes con lógica similar.
Fix: Extraer a una función utilitaria compartida: verifyShopifyHmac(rawBody, hmacHeader, secret)
MEDIO HTTP status codes genéricos (mayormente 500) API Design ▼
Todo el backend
La mayoría de errores retornan 500. Faltan códigos apropiados: 409 (conflicto), 422 (validación), 429 (rate limit), 404 (no encontrado).
Fix: Usar status codes HTTP estándar consistentemente según el tipo de error.
MEDIO Sin versionamiento de API API Design ▼
cloud-run-api/src/routes/
Los endpoints no tienen prefijo de versión (ej: /v1/). Cambios breaking en el futuro serán difíciles de manejar.
Fix: Agregar versionamiento: /api/v1/...
MEDIO 391 console.log en producción (backend) Logging ▼
Múltiples archivos del backend
Console.log/warn/error usados para logging en producción. No son estructurados, no tienen niveles, y dificultan el monitoreo.
Fix: Reemplazar con librería de logging estructurado (pino, winston) integrada con Google Cloud Logging.
BAJO Promesas fire-and-forget sin catch Error Handling ▼
cloud-run-api/src/routes/email/track-events.ts:17-43
Operaciones async fire-and-forget dentro de IIFE sin manejo de errores. Si la promesa rechaza, el error se pierde silenciosamente.
Fix: Agregar .catch(err => console.error(...)) a todas las promesas fire-and-forget.

ALTO N+1 queries en procesamiento de bounces Database ▼
cloud-run-api/src/routes/email/track-events.ts:143-170
Para cada email rebotado, se ejecuta una query individual para buscar el suscriptor. Con 100+ bounces en un webhook, esto genera N+1 queries.
Fix: Batch query: select('id').in('email', recipientEmails) y luego procesar en loop.
MEDIO PDF imports sin lazy loading Bundle Size ▼
src/components/client-portal/BrandBriefView.tsx:15-21
Librerías de PDF (jsPDF, registerPdfFont) importadas globalmente. Aumentan el bundle size para todos los usuarios aunque no generen PDFs.
Fix: Usar React.lazy() o import() dinámico para cargar las librerías de PDF solo cuando se necesiten.
MEDIO Cálculos de bloques normalizados en cada render Memoization ▼
src/components/client-portal/email/EmailBlockEditor.tsx:87-93
La normalización de bloques se calcula en cada render sin useMemo. Con muchos bloques, esto es costoso.
Fix: Envolver en useMemo(() => normalizeBlocks(blocks), [blocks])
MEDIO Scraping de competidores bloquea el endpoint Async ▼
cloud-run-api/src/routes/ai/analyze-brand-research.ts:233-260
El scraping de 6+ competidores espera a que todos terminen antes de responder. Con 10+ segundos por scrape, el endpoint puede hacer timeout.
Fix: Usar Cloud Tasks para procesamiento async. Retornar un job ID y que el cliente haga polling.
MEDIO Inserciones de eventos email una por una Database ▼
cloud-run-api/src/routes/email/track-events.ts:143-226
Cuando un webhook SES trae 100+ bounces, se insertan registros uno por uno en loop.
Fix: Usar batch insert: supabase.from('email_events').insert([...records])
MEDIO Cache in-memory no compartido entre instancias Cache ▼
cloud-run-api/src/routes/email/product-recommendations.ts:84-99
Catálogo de productos cacheado en memoria. En deploy multi-instancia de Cloud Run, cada instancia tiene su propio cache.
Fix: Usar Supabase o Redis para cache compartido con TTL.
BAJO Cache de exchange rates sin invalidación Cache ▼
cloud-run-api/src/routes/shopify/sync-shopify-metrics.ts:26-40
Variable global cachedRates no tiene TTL. Puede servir tasas de cambio desactualizadas indefinidamente.
Fix: Agregar TTL: if (Date.now() - cachedRates.fetchedAt > 3600000) refresh()
