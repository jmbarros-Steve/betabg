Activar la cola para todos los clientes

La feature está deployada hace 1 día y no la usa nadie. Toda la inteligencia (rate limit, retry, smart send time) está muerta.
Esfuerzo2 horas

❌ Problema actual

email_send_settings está vacía. Ningún cliente tiene use_send_queue=true. 100% del envío cae por la ruta directa de manage-campaigns.ts:188-610, que no respeta smart send time, no reintenta, no se recupera de crashes.

✅ Solución

Crear una migration que inserte settings con use_send_queue=true para todos los clientes existentes. Seedear el default en el código. La cola ya funciona — solo nadie la enchufó.

EJEMPLO — Migration mínima supabase/migrations/20260408120000_enable_send_queue_default.sql 
-- 1. Backfill: activar cola para todos los clientes existentes
INSERT INTO email_send_settings (client_id, use_send_queue, rate_limit_per_minute)
SELECT id, true, 60
FROM clients
WHERE NOT EXISTS (
  SELECT 1 FROM email_send_settings WHERE client_id = clients.id
);

-- 2. Default a true para clientes nuevos
ALTER TABLE email_send_settings
  ALTER COLUMN use_send_queue SET DEFAULT true;

-- 3. Trigger para auto-crear settings al crear cliente
CREATE OR REPLACE FUNCTION create_default_email_settings()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO email_send_settings (client_id, use_send_queue)
  VALUES (NEW.id, true);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

2

Smart send time también en la ruta directa

Los clientes que envían sin cola pierden el feature de "enviar a cada subscriber a su mejor hora". Es injusto.
Esfuerzo4 horas

❌ Problema actual

send-queue.ts respeta subscriber.send_time_hour al procesar la cola. Pero manage-campaigns.ts en la ruta directa envía todos los emails al mismo tiempo, ignorando completamente la mejor hora por subscriber.

✅ Solución

Aunque la cola se active (mejora #1), mientras tanto: agrupar subscribers por send_time_hour en la ruta directa y usar Cloud Tasks para programar batches por hora.

EJEMPLO — Agrupar antes de enviar cloud-run-api/src/routes/email/manage-campaigns.ts (rama send directa) 
// ANTES: loop secuencial sin importar la hora
for (const sub of subscribers) {
  await sendSingleEmail({ to: sub.email, ... });
}

// DESPUÉS: agrupar por hora y programar batches
const byHour = subscribers.reduce((acc, sub) => {
  const hour = sub.send_time_hour ?? 10; // default 10am
  (acc[hour] ||= []).push(sub);
  return acc;
}, {} as Record<number, Subscriber[]>);

for (const [hour, batch] of Object.entries(byHour)) {
  const scheduledFor = nextOccurrenceOf(parseInt(hour));
  await scheduleCloudTask({
    payload: { campaignId, subscriberIds: batch.map(s => s.id) },
    scheduledFor,
  });
}

3

Quiet hours por timezone del subscriber, no UTC

Hoy flow-engine.ts usa quiet hours en UTC. Un subscriber chileno con quiet hours 22-08 recibe emails a las 6pm hora local.
Esfuerzo3 horas

❌ Problema actual

La función isInQuietHours() de flow-engine.ts compara contra UTC. Para Chile (UTC-3), las quiet hours configuradas como 22-08 realmente significan 19-05 local. Resultado: spam nocturno legal pero molesto.

✅ Solución

Agregar columna timezone a email_subscribers (default America/Santiago). Convertir quiet hours antes de comparar usando Intl.DateTimeFormat.

EJEMPLO — Conversión correcta cloud-run-api/src/routes/email/flow-engine.ts 
// ANTES: usa UTC ciegamente
function isInQuietHours(quietStart: number, quietEnd: number): boolean {
  const nowUtc = new Date().getUTCHours();
  return nowUtc >= quietStart || nowUtc < quietEnd;
}

// DESPUÉS: respeta timezone del subscriber
function isInQuietHours(
  quietStart: number,
  quietEnd: number,
  timezone = 'America/Santiago'
): boolean {
  const hourLocal = parseInt(
    new Intl.DateTimeFormat('en-US', {
      hour: '2-digit',
      hour12: false,
      timeZone: timezone,
    }).format(new Date())
  );
  return hourLocal >= quietStart || hourLocal < quietEnd;
}

// Migration acompañante:
ALTER TABLE email_subscribers
  ADD COLUMN timezone TEXT DEFAULT 'America/Santiago';

Una vez activada (mejora #1), hay que ver: items stuck, error rate, throughput por minuto, items por cliente.
Esfuerzo6 horas

❌ Problema actual

EJEMPLO — Query base del dashboard cloud-run-api/src/routes/email/queue-health.ts (nuevo) 
export async function queueHealth(c: Context) {
  const [
    statusCounts,
    stuckItems,
    throughputLastHour,
    topClients,
    recentErrors,
  ] = await Promise.all([
    supabase.rpc('email_queue_status_counts'),
    supabase.from('email_send_queue')
      .select('id, processed_at, last_error')
      .eq('status', 'processing')
      .lt('processed_at', tenMinutesAgo()),
    supabase.rpc('email_queue_throughput', { window_minutes: 60 }),
    supabase.rpc('email_queue_top_clients', { limit_n: 10 }),
    supabase.from('email_send_queue')
      .select('id, client_id, last_error, processed_at')
      .not('last_error', 'is', null)
      .order('processed_at', { ascending: false })
      .limit(20),
  ]);
  return c.json({ statusCounts, stuckItems, throughputLastHour, topClients, recentErrors });
}

Alertas automáticas en bounces/complaints masivos

Resend procesa los webhooks pero no avisa cuando hay 50 bounces seguidos. Eso es señal de lista contaminada o dominio quemado.
Esfuerzo3 horas

❌ Problema actual

El handler de Resend webhook actualiza email_subscribers.status a 'bounced' o 'complained' silenciosamente. Si una campaña tiene 10% de bounce (5x el benchmark), nadie se entera hasta que abren analytics.

✅ Solución

Agregar trigger en Postgres que cuente bounces/complaints en ventana de 1h por client_id, y si supera umbral, inserta en steve_alerts y dispara mensaje a Slack/WhatsApp del admin.

EJEMPLO — Trigger en DB supabase/migrations/20260408130000_email_bounce_alerts.sql 
CREATE OR REPLACE FUNCTION alert_on_bounce_spike()
RETURNS TRIGGER AS $$
DECLARE
  recent_bounces INT;
  recent_total   INT;
  bounce_rate    NUMERIC;
BEGIN
  IF NEW.event_type NOT IN ('bounced', 'complained') THEN
    RETURN NEW;
  END IF;

  SELECT COUNT(*) INTO recent_bounces
  FROM email_events
  WHERE client_id = NEW.client_id
    AND event_type IN ('bounced', 'complained')
    AND created_at > NOW() - INTERVAL '1 hour';

  SELECT COUNT(*) INTO recent_total
  FROM email_events
  WHERE client_id = NEW.client_id
    AND created_at > NOW() - INTERVAL '1 hour';

  bounce_rate := (recent_bounces::NUMERIC / NULLIF(recent_total, 0)) * 100;

  IF bounce_rate > 5 AND recent_total >= 20 THEN
    INSERT INTO steve_alerts (client_id, severity, message, payload)
    VALUES (
      NEW.client_id,
      'high',
      format('Bounce rate %.1f%% en última hora (%s/%s)',
        bounce_rate, recent_bounces, recent_total),
      jsonb_build_object('bounces', recent_bounces, 'total', recent_total)
    );
  END IF;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_email_bounce_alert
  AFTER INSERT ON email_events
  FOR EACH ROW EXECUTE FUNCTION alert_on_bounce_spike();
