CRITICAL manage-meta-campaign.ts:1435
No se pueden gestionar campanas para merchants conectados via BM Partner
Imagina que un merchant se conecto a Steve via Leadsie (no OAuth directo). Su token vive en una variable de entorno del servidor, NO en la base de datos. Pero este archivo revisa si hay token en la base de datos antes de continuar — como no lo encuentra, devuelve error 400 y el merchant no puede crear, pausar ni editar campanas.
Codigo actual (malo)
// Linea 1435 — rechaza si no hay token en DB if (!connection.access_token_encrypted || !connection.account_id) { return c.json({ error: 'Missing Meta credentials' }, 400); } // El merchant SUAT nunca pasa de aqui ❌
Como deberia ser
// Solo verificar que la conexion existe if (!connection) { return c.json({ error: 'No Meta connection found' }, 404); } // getTokenForConnection() sabe buscar en DB o en env var const token = await getTokenForConnection(supabase, connection);
CRITICAL execute-meta-rules.ts:55
Las reglas automaticas nunca se ejecutan para merchants SUAT
Las reglas tipo "si el CPA sube de $5.000, pausar la campana" se revisan cada hora via cron. Pero el cron salta silenciosamente todas las conexiones SUAT con un continue. Ninguna regla automatica funciona para esos merchants.
Escenario real
1. Merchant "Tienda Pepe" se conecta via Leadsie 2. Crea regla: "Si ROAS < 1.0, pausar campana" 3. ROAS baja a 0.3 — campana quemando plata 4. El cron llega a su conexion... 5. if (!conn?.access_token_encrypted) continue; // lo salta 6. La campana sigue quemando plata indefinidamente ❌

HIGH fetch-meta-ad-accounts.ts:189
Un merchant SUAT puede ver las cuentas de TODOS los otros merchants
Cuando se usa un SUAT (System User Access Token), llamar a /me/adaccounts devuelve todas las cuentas del Business Manager de Steve, no solo la del merchant que lo pidio. Es como si un inquilino pudiera ver los contratos de todos los otros inquilinos del edificio.

Merchant A pide sus ad accounts → GET /me/adaccounts con SUAT → Devuelve: cuenta_A, cuenta_B, cuenta_C, cuenta_D... → Merchant A ve los datos de B, C y D 💀 Fix: usar GET /act_{account_id} directamente

CRITICAL flow-webhooks.ts:493
Usuarios que se desuscribieron son re-suscritos automaticamente (GDPR)
Cuando alguien crea un checkout en Shopify, el webhook llama a upsertSubscriber() que siempre pone status: 'subscribed'. Si esa persona ya se habia desuscrito, se le vuelve a suscribir sin su consentimiento. Esto viola el GDPR (derecho a opt-out).
Escenario real
1. Maria se desuscribe de emails → status: 'unsubscribed' 2. Maria compra algo en la tienda Shopify 3. Webhook dispara upsertSubscriber({ status: 'subscribed' }) 4. Maria vuelve a recibir emails sin querer 😡 5. Reclamo GDPR → multa potencial para el merchant
