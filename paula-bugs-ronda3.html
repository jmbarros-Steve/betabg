#61 RLS USING(true) permite a cualquier usuario autenticado leer TODOS los datos WA cross-tenant CRITICAL migration 20260317100003 Data Exposure ▶
Qué pasa

Las tablas wa_twilio_accounts, wa_credits, wa_messages, wa_conversations, wa_campaigns y wa_automations tienen policies SELECT con USING (true). Cualquier usuario autenticado puede leer mensajes, créditos, tokens Twilio encriptados y campañas de TODOS los merchants desde la consola del browser.

Merchant A abre consola del browser → supabase.from('wa_twilio_accounts').select('*') → obtiene SIDs y tokens encriptados de todos los merchants. Combinado con el XOR débil (#66), es full takeover de cuentas Twilio.

#62 Status callback sin validación de firma Twilio — cualquiera puede forjar callbacks CRITICAL status-callback.ts Security ▶
Qué pasa

El endpoint /api/whatsapp/status-callback acepta POST de cualquiera sin validar X-Twilio-Signature. Contrasta con steve-wa-chat.ts que sí valida. Un atacante que conozca un message_sid puede forjar callbacks para marcar mensajes como delivered/failed y manipular contadores de campañas.

Atacante envía POST con MessageSid conocido + MessageStatus=delivered → sistema actualiza sin verificar. Combinado con status=failed, puede triggerear refunds de créditos o corromper analytics.

#63 Password temporal enviada en texto plano por WA y almacenada legible en wa_messages CRITICAL prospect-trial.ts:33,102 Credential Exposure ▶
Qué pasa

La password temporal se genera en línea 33 y se envía por WA en línea 102. Luego se guarda en wa_messages.body (línea 110-119) en texto plano. Con RLS USING(true) (#61), cualquier usuario autenticado puede hacer SELECT body FROM wa_messages WHERE body LIKE '%clave temporal%' y extraer passwords de todos los usuarios provisionados.

User A autenticado → query wa_messages → extrae temp passwords → login como cualquier nuevo usuario antes de que cambien la clave.

#64 wa-action-processor: .order().limit() en UPDATE es ignorado por PostgREST — toma TODAS las acciones CRITICAL wa-action-processor.ts:42-43 Data Corruption ▶
Qué pasa

El claim step usa .update(...).eq('status','pending').limit(10). PostgREST NO soporta .order() ni .limit() en operaciones UPDATE — estos modificadores solo aplican a SELECT. El update se aplica a TODAS las filas matching (todas las pending), no solo 10. El SELECT posterior con .limit(10) toma solo 10, dejando las demás orphaned en status 'processing' para siempre.

500 acciones pending → el cron las reclama TODAS a 'processing'. SELECT trae 10. Las 490 restantes quedan en 'processing' con el batchId pero nadie las procesa → acciones perdidas permanentemente.

#65 wa-action-processor: count de .update() es siempre null — guard de doble-procesamiento es no-op CRITICAL wa-action-processor.ts:74 Race Condition ▶
Qué pasa

El código destructura { count: updatedRows } de un .update(). Pero Supabase JS NO retorna count por defecto — retorna null a menos que uses { count: 'exact' }. Como null === 0 es false, el guard NUNCA se activa. El fix del Bug #55 es completamente inoperante.

Dos instancias de cron reclaman las mismas acciones. El guard updatedRows===0 nunca dispara. Ambas ejecutan → prospect recibe mensajes WA duplicados.

#66 Booking API completamente pública sin auth ni rate limit — spam de calendario ilimitado CRITICAL booking-api.ts:241-457 Security ▶
Qué pasa

bookingConfirm es un endpoint público (sin JWT, sin rate limit, sin CAPTCHA). Un atacante que descubra un seller_id válido puede hacer POST ilimitados creando eventos de Google Calendar con Google Meet. Los seller IDs se exponen en la respuesta de /api/booking/slots/:sellerId.

Script automatizado → POST rápidos con seller_id válido y distintos slot_start → llena el calendario del vendedor con miles de reuniones fake, agota quota de Google Calendar API.

#67 Status callback: update-then-read hace que idempotency guard siempre skip counters CRITICAL status-callback.ts:30-53 Logic Error ▶
Qué pasa

El handler primero ACTUALIZA wa_messages.status al nuevo valor, luego LEE la misma fila para chequear si el status cambió. Como el UPDATE ya corrió, previousStatus siempre equals status. El guard previousStatus === status SIEMPRE dispara y SIEMPRE salta el incremento de counters. delivered_count y read_count de campañas nunca se incrementan.

Campaña enviada a 500 destinatarios. Twilio envía 500 callbacks de delivered. Ninguno incrementa el counter porque el guard siempre ve el status que acaba de escribir. Analytics de campaña permanentemente en 0.

#68 XOR "encryption" de tokens Twilio es trivialmente reversible con known-plaintext HIGH setup-merchant.ts:15-35 Crypto ▶
Qué pasa

XOR con key estática HMAC. encrypt(encrypt(x)) = x. Si un atacante obtiene un par plaintext-ciphertext (creando su propia cuenta merchant), recupera el keystream y descifra TODOS los tokens. Fallback key es 'fallback' si env vars missing → zero security.

Atacante crea merchant → conoce su auth token de Twilio → lee encrypted de Supabase (Bug #61) → XOR → keystream → descifra tokens de todos los merchants → takeover de sub-cuentas Twilio.

Atacante con cuenta free → POST provision con client_id de víctima → crea sub-cuenta Twilio + phone number cargados a Steve. Puede también llamar 'status' con cualquier client_id para enumerar configuraciones.

Prospect envía URL http://rebind.evil.com que DNS-resuelve a 169.254.169.254 → quickScrapeUrl pasa la validación → fetch obtiene credenciales IAM → potencialmente leakeadas en respuesta AI.
