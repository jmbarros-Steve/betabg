La estrategia ganadora

No elegir. Dejar ambos modos listos.

La clave es abstraer el origen de las credenciales detrás de una capa común. El código de negocio (sync, webhooks, analytics) no sabe si está en modo Custom App o App Store — solo pregunta "dame las credenciales para este shop" y un Credential Resolver decide de dónde sacarlas.

Resultado: hoy funcionamos 100% con Custom Apps, y el día que Shopify apruebe la app pública, seteamos 3 env vars y un feature flag. Los dos modos coexisten. Los clientes existentes no se enteran del cambio.

La idea central

Imagina que tu cocina tiene dos fuentes de agua

Una es del grifo municipal (App Store = credenciales globales). La otra es de un estanque propio (Custom App = credenciales por-cliente). Tú NO quieres que cada receta diga "ve al grifo" o "ve al estanque". Quieres un solo caño que te dé agua, y el plomero decide de dónde la saca.

Ese "plomero" en el código se llama Credential Resolver. Es una función única que recibe un shop_domain y devuelve las credenciales necesarias (client_id, client_secret, access_token, webhook_secret). El resto del sistema no sabe ni le importa cómo se obtuvieron.

El Credential Resolver (código real)

Archivo nuevo: cloud-run-api/src/lib/shopify-credentials.ts

Este es el corazón de la arquitectura dual. ~100 líneas. Es la ÚNICA forma en que el resto del código lee credenciales de Shopify. 

// cloud-run-api/src/lib/shopify-credentials.ts
import { getSupabaseAdmin } from './supabase.js';

export type ShopifyMode = 'custom_app' | 'app_store';

export interface ShopifyCredentials {
  mode: ShopifyMode;
  shopDomain: string;
  connectionId: string;
  clientId: string;       // OAuth app id
  clientSecret: string;   // OAuth app secret (también sirve como webhook HMAC)
  accessToken: string;    // Admin API token del cliente
  webhookSecret: string;  // Para validar HMAC de webhooks
}

/**
 * Resuelve credenciales de Shopify para un shop_domain específico.
 * Funciona en ambos modos (Custom App y App Store) de forma transparente.
 *
 * @param shopDomain - ej. "mi-tienda.myshopify.com"
 */
export async function resolveShopifyCredentials(
  shopDomain: string
): Promise<ShopifyCredentials | null> {
  const supabase = getSupabaseAdmin();

  // 1. Buscar la conexión
  const { data: connection } = await supabase
    .from('platform_connections')
    .select('*')
    .eq('platform', 'shopify')
    .eq('shop_domain', shopDomain)
    .eq('is_active', true)
    .maybeSingle();

  if (!connection) return null;

  // 2. Desencriptar el access_token (siempre vive en la conexión)
  const { data: accessToken } = await supabase
    .rpc('decrypt_platform_token', {
      encrypted: connection.access_token_encrypted,
    });

  // 3. Decidir de dónde sacar client_id y client_secret
  const mode: ShopifyMode = connection.connection_mode || 'custom_app';

  if (mode === 'custom_app') {
    // ── Modo Custom App: credenciales por-conexión ──
    const { data: clientSecret } = await supabase
      .rpc('decrypt_platform_token', {
        encrypted: connection.shopify_client_secret_encrypted,
      });

    return {
      mode: 'custom_app',
      shopDomain,
      connectionId: connection.id,
      clientId: connection.shopify_client_id,
      clientSecret,
      accessToken,
      // En Custom Apps, el webhook HMAC se firma con el client_secret
      webhookSecret: clientSecret,
    };
  }

  // ── Modo App Store: credenciales globales desde Cloud Run env ──
  const globalClientId = process.env.SHOPIFY_CLIENT_ID;
  const globalClientSecret = process.env.SHOPIFY_CLIENT_SECRET;
  const globalWebhookSecret = process.env.SHOPIFY_WEBHOOK_SECRET;

  if (!globalClientId || !globalClientSecret || !globalWebhookSecret) {
    throw new Error(
      'App Store mode activo pero faltan env vars globales'
    );
  }

  return {
    mode: 'app_store',
    shopDomain,
    connectionId: connection.id,
    clientId: globalClientId,
    clientSecret: globalClientSecret,
    accessToken,
    webhookSecret: globalWebhookSecret,
  };
}

/**
 * Lee el modo global del sistema desde env.
 * Útil para decidir qué UI mostrar (wizard vs 1-click vs ambos).
 */
export function getShopifyMode(): 'custom' | 'appstore' | 'both' {
  const mode = (process.env.SHOPIFY_MODE || 'custom').toLowerCase();
  if (mode === 'appstore') return 'appstore';
  if (mode === 'both') return 'both';
  return 'custom';
}

Clave del diseño: la función resolveShopifyCredentials recibe shop_domain y no connectionId. ¿Por qué? Porque los webhooks llegan con el header X-Shopify-Shop-Domain pero NO con el connectionId. Usar shop_domain nos permite usar el mismo resolver tanto en routes del backend como en webhooks de edge functions.

Cómo se usa en un webhook handler

// shopify-fulfillment-webhooks/index.ts (refactor)
const shopDomain = req.headers.get('x-shopify-shop-domain') || '';
const creds = await resolveShopifyCredentials(shopDomain);

if (!creds) {
  return new Response('Shop not registered', { status: 404 });
}

// Verifica HMAC con el secret correcto (custom o global, transparente)
if (!verifyWebhookHmac(rawBody, hmacHeader, creds.webhookSecret)) {
  return new Response('Unauthorized', { status: 401 });
}

// ... procesa el webhook como siempre

Nota: las edge functions de Supabase (Deno) no pueden importar directamente del backend Node.js. Hay que duplicar este resolver en supabase/functions/_shared/shopify-credentials.ts (versión Deno). Es ~30 líneas, no es drama, y asegura que ambos mundos comparten la lógica.

La migración de base de datos

Nueva columna en platform_connections

Marcamos cada conexión Shopify con su modo. Las conexiones existentes (las que tienes hoy con tus clientes de prueba) se marcan como 'custom_app' por default. 

-- supabase/migrations/20260408_shopify_dual_mode.sql

ALTER TABLE platform_connections
ADD COLUMN IF NOT EXISTS connection_mode text
  NOT NULL DEFAULT 'custom_app'
  CHECK (connection_mode IN ('custom_app', 'app_store'));

COMMENT ON COLUMN platform_connections.connection_mode IS
  'Cómo fue conectado: custom_app (cliente crea su propia app) o app_store (OAuth público)';

-- Índice para queries por modo (útil en el dashboard de admin)
CREATE INDEX IF NOT EXISTS idx_platform_connections_mode
  ON platform_connections(platform, connection_mode)
  WHERE platform = 'shopify';

-- Las conexiones existentes ya tienen shopify_client_id/secret por-cliente,
-- así que quedan marcadas como 'custom_app' automáticamente via DEFAULT.

-- El día del switch a App Store, los clientes nuevos se marcarán
-- como 'app_store' desde el callback de OAuth público.

¿Y los clientes híbridos? Un cliente NO puede tener dos conexiones activas al mismo Shopify (una custom y otra app store). El resolver siempre retorna la primera is_active = true. Si en el futuro quieres migrar a un cliente de custom a app store, solo desactivas la conexión vieja y la reemplazas.

Reglas de oro (invariantes del sistema dual)

1. Ningún archivo (excepto el resolver) puede leer SHOPIFY_CLIENT_ID directamente

Si un dev lo hace, rompe la dualidad. Enforced por lint rule o grep en CI.

2. Los webhooks siempre buscan credenciales por shop_domain, nunca por connectionId

Porque los headers de Shopify traen shop_domain, no el UUID interno.

3. connection_mode es inmutable tras crearse

Si un cliente quiere migrar, se crea una nueva conexión y se desactiva la vieja. Nunca se edita el modo.

4. El feature flag SHOPIFY_MODE solo controla UI, nunca backend

Cambiar el flag no debe afectar las conexiones existentes. Solo decide qué flujo ve un cliente nuevo.

5. El resolver devuelve null si la conexión no existe o está inactiva

Los callers siempre deben manejar el caso null. Nunca asumir que hay credenciales.
