Decisión arquitectural

Vamos solo con Custom Apps. Esto cambia todo.

JM decidió: por ahora no publicamos en Shopify App Store. Cada cliente crea su propia Custom App en Shopify Partners y nos pega las credenciales en el wizard. Esta decisión elimina 3 problemas, pero crea 4 nuevos y vuelve obsoletas ~1.200 líneas de código.

¿Qué significa "Custom Apps Only"?

ANTES (App Store público)

Las credenciales eran globales

  • • Una sola app de Steve Ads instalada por todos los clientes
  • • 3 env vars globales en Cloud Run (CLIENT_ID, SECRET, WEBHOOK_SECRET)
  • • Un solo webhook secret firmaba todos los webhooks
  • • Aprobación de Shopify obligatoria (2-4 semanas)
  • • Cliente da 1 clic → instalado
AHORA (Custom Apps)

Cada cliente tiene SU propia app

  • • Una app por cliente creada en su Shopify Partners
  • • Credenciales por conexión guardadas encriptadas en DB
  • • Cada cliente tiene su webhook secret distinto
  • • Sin aprobación de Shopify, podemos vender mañana
  • • Cliente sigue 6 pasos en el wizard

El insight clave: el código actual fue diseñado para el modelo App Store público. Hay piezas que "funcionaban por casualidad" porque los clientes de prueba usaban las mismas credenciales que JM puso en su Shopify Partners. Cuando lleguen 5 clientes reales con 5 Custom Apps distintas, varias cosas se rompen.

Lo que YA NO es problema

Blocker #0 — Faltan 3 env vars

RESUELTO

¿Por qué ya no importa?

Las 3 variables (SHOPIFY_CLIENT_ID, SHOPIFY_CLIENT_SECRET, SHOPIFY_WEBHOOK_SECRET) eran necesarias porque había una sola app pública firmando todo. En el modelo Custom App, cada cliente genera SUS propias credenciales en su Shopify Partners y las pega en el wizard. Las credenciales viven encriptadas en platform_connections.

Imagina que...

Antes te preocupaba conseguir la llave maestra del edificio. Ahora cada oficina tiene su propia chapa y cada inquilino trae su propia llave. La llave maestra ya no existe ni hace falta.

Acción: No hacer nada. Las env vars pueden quedar vacías para siempre. Eventualmente las podemos eliminar de la lista de "vars pendientes" en CLAUDE.md.

Esperar aprobación de Shopify

RESUELTO

Las apps públicas requieren review de Shopify que toma 2-4 semanas y exige cumplir ~80 requisitos (políticas de privacidad, GDPR, performance, screenshots, video demo, etc.). Las Custom Apps no requieren ninguna aprobación. Podemos cobrar plata mañana.

Acción: Olvidarse del proceso de Shopify Partners review.

S5 · Fallback peligroso a SHOPIFY_CLIENT_SECRET

RESUELTO (a la fuerza)

El fallback a SHOPIFY_CLIENT_SECRET en el handler de webhooks ya no aplica porque esa variable no existe. Pero ojo: este "fix" es ilusorio, porque ahora NINGÚN secret global sirve y aparece un nuevo blocker (ver N1).

Lo NUEVO que ahora es crítico

N1 · Los webhooks NO se pueden validar con un secret global

BLOCKER #1

supabase/functions/shopify-fulfillment-webhooks/index.ts:288-306
supabase/functions/shopify-gdpr-webhooks/index.ts:79-83

¿Qué pasa?

Los handlers de webhook leen un único SHOPIFY_WEBHOOK_SECRET global para verificar el HMAC. Pero en Custom Apps, cada app del cliente firma con SU propio secret. Si el Cliente A manda un webhook firmado con su secret, y nosotros lo verificamos con el secret del Cliente B, falla siempre.

Imagina que...

Eres notario. Recibes documentos firmados de varios clientes. Cada cliente firma con su propia rúbrica. Tú tienes en tu cajón una sola muestra de firma que comparas con todos. Como ninguna firma coincide con tu única muestra, rechazas TODOS los documentos.

Ejemplo concreto

Cliente A configura su Custom App con secret aaa-secret. Cliente B con bbb-secret. Llega un webhook de "orders/fulfilled" del Cliente A. El handler intenta verificarlo con SHOPIFY_WEBHOOK_SECRET (vacío o random). El HMAC falla. El webhook se rechaza con 401. El pedido nunca se procesa. Felipe (W2) sigue mostrando "stock disponible" para un producto que ya se vendió.

El código actual (malo)

// Lee UN secret global, no por cliente
const shopifySecret = Deno.env.get('SHOPIFY_WEBHOOK_SECRET')
                   || Deno.env.get('SHOPIFY_CLIENT_SECRET');

if (!verifyWebhookHmac(rawBody, hmacHeader, shopifySecret)) {
  return new Response('Unauthorized', { status: 401 });
}

El fix (bueno)

// Buscar el secret específico del cliente usando el shop_domain del header
const shopDomain = req.headers.get('x-shopify-shop-domain') || '';

const { data: connection } = await supabaseAdmin
  .from('platform_connections')
  .select('shopify_client_secret_encrypted')
  .eq('platform', 'shopify')
  .eq('shop_domain', shopDomain)
  .eq('is_active', true)
  .maybeSingle();

if (!connection) {
  return new Response('Shop not registered', { status: 404 });
}

// Desencriptar el secret del cliente
const { data: clientSecret } = await supabaseAdmin
  .rpc('decrypt_platform_token', { encrypted: connection.shopify_client_secret_encrypted });

if (!verifyWebhookHmac(rawBody, hmacHeader, clientSecret)) {
  return new Response('Unauthorized', { status: 401 });
}

OJO: En Custom Apps, el "webhook secret" suele ser el mismo API secret key de la app. Eso ya lo tenemos guardado como shopify_client_secret_encrypted. Hay que verificarlo con la primera Custom App real.

N2 · No hay registro automático de webhooks en Custom Apps

BLOCKER #2

¿Qué pasa?

El registro automático de los 7 webhooks (orders/create, fulfilled, GDPR, etc.) solo se ejecuta dentro del flujo OAuth público, en shopify-oauth-callback.ts. En el wizard de Custom App, nadie llama a esa función. Resultado: después del wizard, los webhooks NO existen y los handlers no reciben absolutamente nada.

Imagina que...

Contratas un servicio de mensajería. Te firman el contrato pero nunca pasas tu dirección por WhatsApp. Los mensajeros existen, los paquetes existen, pero no saben dónde dejarlos. Tú esperas paquetes que nunca llegan y no entiendes por qué.

Ejemplo concreto

Cliente A termina el wizard. Su tienda Shopify queda conectada a Steve Ads. Hace una venta. Shopify NO envía un webhook a Steve Ads porque nadie le dijo a Shopify "manda eventos a esta URL". El cliente ve su venta en Shopify pero NO en Steve Ads. Felipe (W2) reporta revenue desactualizado. Ignacio (W17) dice "no tengo data".

El fix

Después de que el wizard guarda el access_token, llamar inmediatamente a la Admin API del cliente para registrar los 7 webhooks contra las URLs de nuestras Edge Functions: 

// Después del wizard, en store-shopify-token.ts:
async function registerWebhooks(shopDomain, accessToken) {
  const topics = [
    'orders/create', 'orders/fulfilled', 'orders/cancelled',
    'app/uninstalled',
    'customers/data_request', 'customers/redact', 'shop/redact',
  ];
  for (const topic of topics) {
    await fetch(`https://${shopDomain}/admin/api/2024-01/webhooks.json`, {
      method: 'POST',
      headers: {
        'X-Shopify-Access-Token': accessToken,
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({
        webhook: {
          topic,
          address: `https://zpswjccsxjtnhetkkqde.supabase.co/functions/v1/shopify-${topic.includes('redact') || topic.includes('data_request') ? 'gdpr' : 'fulfillment'}-webhooks`,
          format: 'json',
        },
      }),
    });
  }
}

N3 · El wizard hace polling sin timeout

BLOCKER #3

src/components/client-portal/ShopifyCustomAppWizard.tsx:71-94

¿Qué pasa?

El paso 5 del wizard hace polling cada 3 segundos esperando que aparezca el access_token_encrypted. Si nunca aparece (porque el cliente abandonó, dio mal el secret, o el OAuth falló), el polling sigue para siempre hasta que el usuario cierre el browser.

Imagina que...

Vas al Registro Civil a buscar tu carnet. Te dicen "espera ahí". Nunca te dicen cuánto. Pasan 5, 10, 30 minutos. No hay reloj, no hay aviso. Tienes que decidir tú cuándo darte por vencido. La mitad de la gente se va frustrada pensando "no me lo van a dar".

Ejemplo concreto

Cliente pega bien sus credenciales (paso 4). Hace clic en "Instalar" (paso 5). Pero su Custom App tiene los scopes mal configurados. Shopify rechaza el OAuth con error. El callback nunca guarda el token. El wizard sigue mostrando "Esperando instalación..." durante 20 minutos. El cliente abandona. Tú no te enteras de que falló.

El fix

useEffect(() => {
  if (step !== 5 || !open) return;

  const startTime = Date.now();
  const TIMEOUT_MS = 5 * 60 * 1000; // 5 minutos máximo

  pollRef.current = setInterval(async () => {
    // ... fetch ...

    // Timeout: dar feedback útil
    if (Date.now() - startTime > TIMEOUT_MS) {
      clearInterval(pollRef.current!);
      setError('La instalación tardó demasiado. Verifica que los scopes estén bien configurados o reinicia el wizard.');
      setPolling(false);
    }
  }, 3000);
}, [step, open]);

N4 · No hay validación de credenciales antes de guardar

BLOCKER #4

cloud-run-api/src/routes/shopify/store-shopify-credentials.ts:55-104

¿Qué pasa?

El endpoint store-shopify-credentials guarda lo que le mandes sin verificar que las credenciales sean válidas. Si el cliente pega un Client ID o Secret incorrecto (typo, copy-paste mal), nadie se da cuenta hasta el paso 5 del wizard cuando OAuth falla silenciosamente.

Imagina que...

Vas al banco a abrir cuenta. Te piden tu RUT. Pones cualquier número. El cajero te dice "perfecto, cuenta abierta". 3 días después, cuando intentas hacer la primera transacción, descubres que el RUT no era válido y la cuenta nunca existió.

El fix

Antes de guardar las credenciales, hacer una llamada de prueba a la Admin API del cliente con esas credenciales. Si responde 401/403, devolver error inmediatamente: 

// Validar credenciales contra Shopify antes de guardar
const testResponse = await fetch(
  `https://${normalizedDomain}/admin/oauth/access_scopes.json`,
  {
    headers: {
      'X-Shopify-Access-Token': 'test', // No funcionará pero validará el dominio
    },
  }
);

if (testResponse.status === 404) {
  return c.json({ error: 'El dominio Shopify no existe o está mal escrito' }, 400);
}

// (validación del API secret se hace en el siguiente paso del OAuth)
