Problema

Los 16 checks de tipo data_quality estaban mal implementados:

  • Checks 15, 16, 18, 19, 20 — tenían validaciones que no coincidían con la descripción real en la DB (se inventaron en el primer deploy)
  • Checks 7, 10, 29, 30, 37, 38, 39, 40, 43, 49, 50 — caían al fallback genérico que solo contaba nulls (inútil)

Restricciones descubiertas: shopify_orders NO EXISTE, shopify_products no tiene columna price (tiene price_min/price_max), el campo es last_sync_at (no last_synced_at).

Los 5 checks que tenían validación INCORRECTA

#15 Shopify CORREGIDO No hay productos en Supabase que no existen en Shopify

QUE HACIA ANTES

Validaba platform_metrics: que metric_type no sea null, que revenue no sea negativo, que metric_value sea un número válido. Nada que ver con productos.

QUE HACE AHORA

  1. Query shopify_products.shopify_product_id WHERE client_id = merchant
  2. Si vacío → pass (nada que validar)
  3. Llama Shopify API GET /products.json?ids={ids}&fields=id en batches de 100
  4. Compara: si algún ID de Supabase no existe en Shopify → fail
Token ShopifyShopify APIshopify_products
#16 Shopify CORREGIDO No hay productos en Shopify que faltan en Supabase

QUE HACIA ANTES

Validaba campaign_metrics: CTR > 100%, CPC negativo, impressions < clicks. Nada que ver con Shopify.

QUE HACE AHORA

  1. Llama Shopify API GET /products/count.json → obtiene total
  2. Query count de shopify_products WHERE client_id = merchant
  3. Si Shopify tiene más productos que Supabase → fail con la diferencia
Token ShopifyShopify APIshopify_products
#18 Shopify CORREGIDO Todas las imágenes de productos cargan (no 404)

QUE HACIA ANTES

Verificaba shopify_products buscando price=0 con status active y títulos vacíos. Además usaba columna price que no existe (la tabla tiene price_min/price_max).

QUE HACE AHORA

  1. Query shopify_products.image_url WHERE not null, limit 10 (configurable)
  2. Si DB vacía: fallback a Shopify API GET /products.json?fields=id,images
  3. HEAD request a cada URL de imagen
  4. Si alguna retorna 404/410 → fail
Token ShopifyShopify API (fallback)shopify_products
#19 Shopify CORREGIDO Revenue de hoy > 0 si merchant normalmente vende

QUE HACIA ANTES

Validaba tabla email_events: fechas futuras y open_rate en metadata. Nada que ver con revenue ni Shopify.

QUE HACE AHORA

  1. Calcula todayStart = hoy 00:00:00
  2. Shopify API: GET /orders.json?created_at_min={today}&status=any&fields=id,total_price
  3. Suma total_price de todos los orders
  4. Si revenue = $0 y min_daily_avg > 0 → fail
Token ShopifyShopify Orders API
#20 Shopify CORREGIDO Fecha del order más reciente es de hoy

QUE HACIA ANTES

Validaba tabla clients: emails vacíos, nombres vacíos, emails duplicados. Nada que ver con orders.

QUE HACE AHORA

  1. Shopify API: GET /orders.json?limit=1&order=created_at+desc&status=any
  2. Si no hay orders → skip
  3. Calcula horas desde el último order
  4. Si > max_hours_old (default 24h) → fail
Token ShopifyShopify Orders API

Los 11 checks que caían al fallback genérico

Todos estos caían al default: del switch, que ejecutaba validateGeneric(). Esa función solo contaba campos null en una tabla genérica — completamente inútil para lo que debían validar.

FALLBACK GENERICO ELIMINADO
async function validateGeneric(supabase, check, merchant) {
  const table = check.check_config?.table || check.check_config?.data_source;
  if (!table) return { invalid_count: 0, total_count: 0, sample_errors: ['No table configured'] };

  // Solo contaba campos null... completamente inutil
  for (const row of data) {
    const nullFields = Object.entries(row)
      .filter(([key, val]) => val === null && ...)
    if (nullFields.length > 3) errors.push(...);
  }
}
#7 All NUEVO No hay platform_connections duplicadas por merchant + plataforma

Como funciona

  1. Query platform_connections WHERE is_active = true, select client_id, platform
  2. Agrupa en JS por client_id::platform
  3. Si algún grupo tiene >1 registro → fail con detalle de duplicados
platform_connections
#10 All NUEVO Ninguna conexión lleva > 24hrs sin sync exitoso

Como funciona

  1. Query platform_connections WHERE is_active = true, select last_sync_at
  2. Calcula cutoff = ahora - max_hours (config, default 24h)
  3. Filtra conexiones donde last_sync_at < cutoff ó es null (nunca sincrono)
  4. Si hay alguna stale → fail con lista de conexiones atrasadas
platform_connections.last_sync_at

#38 Klaviyo NUEVO Bounce rate < 1%

Como funciona

  1. Query platform_metrics WHERE metric_type = 'bounce_rate' ORDER BY metric_date DESC LIMIT 1
  2. Si metric_value > max_bounce (config, default 0.01 = 1%) → fail
  3. Si no hay datos → skip
platform_metrics
#39 Klaviyo NUEVO Spam rate < 0.1%

Como funciona

  1. Query platform_metrics WHERE metric_type = 'spam_rate' ORDER BY metric_date DESC LIMIT 1
  2. Si metric_value > max_spam (config, default 0.001 = 0.1%) → fail
  3. Si no hay datos → skip
platform_metrics
#40 Klaviyo NUEVO Deliverability: SPF + DKIM verificados

Como funciona

  1. Query platform_metrics WHERE metric_type = 'spf_pass' (más reciente)
  2. Query platform_metrics WHERE metric_type = 'dkim_pass' (más reciente)
  3. Si no hay datos de ninguno → skip
  4. Si alguno no está en "true"/"1" → fail con detalle
platform_metrics (spf_pass, dkim_pass)
#43 Steve Chat NUEVO Steve Chat responde en español, no inglés

Como funciona (sin LLM)

  1. POST interno a /api/ai/steve-chat con "¿Cuáles son las métricas de hoy?"
  2. Cuenta palabras inglesas: "the", "and", "your", "campaign", etc.
  3. Cuenta palabras españolas: "las", "los", "del", "para", "métricas", etc.
  4. Si ratio inglés > español y EN ≥ 3 → fail

  3. Verifica que al menos 1 brief contenga el nombre de un merchant real
  4. Si ninguno menciona datos reales → fail
