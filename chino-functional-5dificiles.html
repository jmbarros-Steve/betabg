#77 Steve Chat: prospect flow detecta URLs y hace scraping
steve_chat medium
Lo que encontre en el codigo
Ya existe: Funcion quickScrapeUrl() en steve-wa-chat.ts (linea 90).
Cuando un prospect envia un mensaje con URL, Steve la detecta con regex y hace scraping automatico.
El flow: detecta URL con regex → hace GET a la pagina → extrae titulo, meta description, H1, H2, productos, JSON-LD → inyecta como contexto para Claude.
Esto vive en el webhook de WhatsApp (POST /api/whatsapp/steve-wa-chat), que es un webhook de Twilio, no un endpoint REST normal.
Como probarlo (paso a paso)
1. Simular mensaje con URL
→
2. Llamar quickScrapeUrl
→
3. Verificar datos
// No podemos llamar el webhook de Twilio directamente (requiere firma Twilio). // Pero podemos replicar la logica: scrape una URL publica y verificar resultado. // Opcion A: Testear el scraping directamente const testUrl = 'https://steve.cl'; const res = await fetch(testUrl, { headers: { 'User-Agent': 'Mozilla/5.0 (compatible; SteveBot/1.0)' }, signal: AbortSignal.timeout(5000) }); const html = await res.text(); // Verificar que podemos extraer info basica const hasTitle = html.match(/<title[^>]*>([^<]+)<\/title>/i); const hasH1 = html.match(/<h1[^>]*>/i); // Si tiene titulo o H1 → el scraping funcionaria → PASS // Opcion B: Testear via steve-chat API (modo estrategia) const chatRes = await fetch(`${API}/api/steve-chat`, { method: 'POST', headers: steveHeaders(), body: JSON.stringify({ client_id: merchant.client_id, message: 'analiza esta tienda: https://steve.cl', mode: 'estrategia' }) }); // Verificar que la respuesta menciona algo de la pagina (titulo, productos, etc) // No un "no puedo acceder" generico
La mejor estrategia
Opcion A es mas confiable: Hacer un GET directo a una URL conocida (steve.cl o cualquier sitio publico) y verificar que el scraping extrae titulo/descripcion. Es determinista y no depende de AI.

Opcion B es mas completa pero depende de que Claude genere una respuesta que mencione los datos del sitio, lo cual no es 100% determinista.
FACIL — Solo es un GET a una URL publica
El "scraping" es simplemente un fetch + regex de titulo/H1/meta. No necesita Twilio ni WhatsApp para testear.
#78 Steve Chat: multi-brain (investigator, strategist, conversationalist) responde
steve_chat medium
Lo que encontre en el codigo
Ya existe: Archivo cloud-run-api/src/lib/steve-multi-brain.ts
3 funciones exportadas: runInvestigator(), runStrategist(), runConversationalist()
Pipeline:
1. Investigator (Haiku, ~1s): Carga datos pre-scrapeados del prospect, competitor insights, sales learnings

El multi-brain se usa automaticamente en el webhook de WhatsApp para prospects (numeros desconocidos). Si falla, tiene fallback a single-call con Claude Sonnet.

3. Verificar output

#79 Steve Chat: audio transcription funciona para audios WhatsApp
steve_chat medium
Lo que encontre en el codigo
Ya existe: Archivo cloud-run-api/src/lib/audio-transcriber.ts (113 lineas)
Exporta: isSupportedAudio() y transcribeAudio()
Como funciona:
1. Twilio envia el audio como MediaUrl0 en el webhook
2. isSupportedAudio() verifica formato (ogg, mp3, amr, mp4, wav, webm, aac)
3. transcribeAudio() descarga el audio de Twilio (con auth Basic) → envia a OpenAI Whisper API → retorna texto en espanol
4. El texto transcrito se inyecta como [Audio transcrito]: {texto} en el mensaje
Dependencias: OPENAI_API_KEY (ya configurada en Cloud Run), TWILIO_ACCOUNT_SID, TWILIO_AUTH_TOKEN
Como probarlo (paso a paso)
1. Verificar Whisper API
→
2. Enviar audio de prueba
→
3. Verificar transcripcion
// No necesitamos un audio REAL de WhatsApp. Podemos testear Whisper directamente. // Paso 1: Generar un mini audio con texto conocido // Usamos un archivo de audio de prueba hosteado (o generado con TTS) // Opcion A: Testear que Whisper API responde (sin audio real) const OPENAI_KEY = process.env.OPENAI_API_KEY; if (!OPENAI_KEY) return { result: 'skip', error_message: 'OPENAI_API_KEY not set' }; // Crear un WAV minimo valido (1 segundo de silencio) para verificar que Whisper responde const silentWav = createMinimalWav(); // 44 bytes header + data const formData = new FormData(); formData.append('file', new Blob([silentWav], { type: 'audio/wav' }), 'test.wav'); formData.append('model', 'whisper-1'); formData.append('language', 'es'); const whisperRes = await fetch('https://api.openai.com/v1/audio/transcriptions', { method: 'POST', headers: { Authorization: `Bearer ${OPENAI_KEY}` }, body: formData, }); // Si Whisper responde 200 (aunque sea vacio para silencio) → la API funciona → PASS // Si responde 401/403 → API key mala → FAIL // Si responde 5xx → servicio caido → ERROR // Opcion B: Testear con audio TTS generado // Usar OpenAI TTS para generar audio → enviar a Whisper → verificar roundtrip const ttsRes = await fetch('https://api.openai.com/v1/audio/speech', { method: 'POST', headers: { Authorization: `Bearer ${OPENAI_KEY}`, 'Content-Type': 'application/json' }, body: JSON.stringify({ model: 'tts-1', voice: 'nova', input: 'hola mundo' }) }); const audioBlob = await ttsRes.blob(); // Ahora transcribir const form2 = new FormData(); form2.append('file', audioBlob, 'test.mp3'); form2.append('model', 'whisper-1'); form2.append('language', 'es'); const result = await fetch('https://api.openai.com/v1/audio/transcriptions', { ... }); const text = await result.text(); // Si text contiene "hola" o "mundo" → roundtrip funciona → PASS
Cual opcion usar?
Opcion A (rapida): Solo verificar que Whisper API acepta la request. No necesita audio real. Confirma que la API key funciona y el servicio responde. ~1 segundo.

Opcion B (completa): Generar audio con TTS → transcribir con Whisper → verificar que el texto sale correcto. Confirma el roundtrip completo. ~3-5 segundos. Cuesta ~$0.001 por test.

Recomiendo Opcion B: Es el test mas completo y cuesta casi nada. Si "hola mundo" pasa por TTS y vuelve como "hola mundo" en Whisper, sabemos que la transcripcion funciona end-to-end.
FACIL — TTS + Whisper roundtrip, sin WhatsApp
No necesitamos Twilio ni WhatsApp real. Generamos audio con OpenAI TTS, lo transcribimos con Whisper, y verificamos que el texto es correcto. Todo via API, sin hardware.
#80 Steve Chat: image vision funciona para imagenes WhatsApp
steve_chat medium
Lo que encontre en el codigo
Ya existe: Funcion downloadImageForVision() en steve-wa-chat.ts (linea 48)
Descarga imagen de Twilio, convierte a base64, la envia a Claude como contenido multimodal.
Como funciona en el flujo real:
1. Twilio envia MediaUrl0 + MediaContentType0
2. downloadImageForVision() descarga la imagen (auth con Twilio creds)
3. La imagen se convierte a base64
4. Se inyecta como content multimodal type: "image" en el mensaje a Claude
5. Claude Sonnet "ve" la imagen y responde describiendo productos, sugiriendo como usarla en ads, etc.
Formatos soportados: JPEG, PNG, GIF, WebP. Limite: 5MB.
Como probarlo (paso a paso)
1. Descargar imagen test
→
2. Enviar a Claude vision
→
3. Verificar descripcion
// No necesitamos Twilio. Usamos una imagen publica y la enviamos a Claude vision. // 1. Descargar una imagen de prueba conocida const imgRes = await fetch('https://picsum.photos/200'); // imagen random pequeña const imgBuffer = Buffer.from(await imgRes.arrayBuffer()); const base64 = imgBuffer.toString('base64'); // 2. FACIL — Imagen publica + Claude vision API
Descargar cualquier imagen, enviar a Claude Haiku con vision, verificar que describe algo. Cuesta ~$0.001 por test. Sin Twilio, sin WhatsApp.
