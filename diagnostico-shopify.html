BLOQUEADOR Severidad máxima

Blocker #0 — La integración con Shopify está 100% MUERTA

Sin esto arreglado, todo lo demás da igual.

¿Qué pasa?

Faltan 3 contraseñas (env vars) en el servidor de Cloud Run que son obligatorias para que Shopify nos deje hablar con su API:

  • ❌ SHOPIFY_CLIENT_ID
  • ❌ SHOPIFY_CLIENT_SECRET
  • ❌ SHOPIFY_WEBHOOK_SECRET

Imagina que...

Tienes una pizzería y quieres recibir pedidos por DiDi Food. DiDi te dice: "toma estas 3 llaves, mete una en la puerta del local, otra en la caja registradora y otra en el teléfono". Si no metes las 3, DiDi no te manda un solo pedido. Tu pizzería puede estar abierta, con horno encendido y empleados esperando, pero sin las 3 llaves no entra plata.

Ejemplo concreto en Steve Ads

Hoy mismo, 7 de abril:

Si un cliente nuevo entra a Steve Ads y hace clic en "Conectar Shopify desde el App Store", la pantalla queda cargando para siempre. El cliente piensa que Steve Ads está roto, cierra el browser, y no vuelve. Acabas de perder un usuario por falta de 3 strings de texto.

¿Hay workaround?

Sí: el wizard "Custom App" de la pestaña Shopify SÍ funciona porque las credenciales las pone el cliente manualmente. Pero es lento, requiere que el cliente sepa entrar al admin de Shopify Partners, y nadie quiere hacer 6 clics cuando esperaba 1.

¿Cómo se arregla?

gcloud run services update steve-api \
  --region=us-central1 \
  --project=steveapp-agency \
  --update-env-vars \
    SHOPIFY_CLIENT_ID=xxx,\
    SHOPIFY_CLIENT_SECRET=yyy,\
    SHOPIFY_WEBHOOK_SECRET=zzz

OJO: Las credenciales se sacan de tu cuenta de Shopify Partners. Si no tienes la app creada en Partners, primero hay que crearla y luego pedir aprobación a Shopify (puede tomar 2-4 semanas).

Críticos — Agujeros de seguridad

S1 · Endpoint público sin contraseña

CRÍTICO

supabase/functions/shopify-data-verify/index.ts:18-217

¿Qué pasa?

Hay un endpoint en internet que cualquiera puede llamar sin login, sin token, sin nada. Se supone que solo lo usa nuestro cron interno, pero olvidamos ponerle una contraseña.

Imagina que...

Tienes una bodega con un control remoto que abre el portón. Pones el control en una mesa en la calle con un cartel que dice "solo para empleados". Cualquier peatón puede agarrar el control, abrir el portón y entrar a husmear.

Ejemplo concreto

Un competidor descubre la URL del endpoint. Hace un script que la llama 10.000 veces por minuto. Cada llamada gasta nuestro cupo de Shopify (rate limit), envenena la tabla qa_log con basura, y nos manda spam por WhatsApp porque cree que hay errores.

El código actual (malo)

// Empieza el endpoint... y ya. No hay check de auth.
serve(async (req) => {
  const { connectionId } = await req.json();
  // ... empieza a comparar datos sin preguntar quién eres
});

El fix (bueno)

serve(async (req) => {
  // PRIMERO chequea la contraseña secreta del cron
  if (req.headers.get('x-cron-secret') !== Deno.env.get('CRON_SECRET')) {
    return new Response('forbidden', { status: 403 });
  }
  // ... ahora sí, hace su trabajo
});

Tiempo estimado del fix: 5 minutos.

S2 · Cualquier usuario puede crear cupones en tiendas ajenas

CRÍTICO

supabase/functions/create-shopify-discount/index.ts:51-57

¿Qué pasa?

El endpoint que crea cupones de descuento no verifica que el usuario logueado sea el dueño de la tienda Shopify a la que le va a meter el cupón.

Imagina que...

Vas a un cajero automático. Insertas tu tarjeta. El cajero te pregunta "¿de qué cuenta sacas?" y te da una lista con todas las cuentas de Chile. Eliges la del vecino, sacas $500.000, y el cajero te los entrega sin chistar.

Ejemplo concreto

Cliente A (que vende muebles) y Cliente B (que vende ropa) están ambos en Steve Ads. Cliente A descubre el ID de la conexión Shopify de Cliente B (es un UUID, pero igual). Crea un cupón TODOGRATIS-100 que da 100% de descuento en todos los productos de Cliente B. Lo publica en redes. Cliente B pierde su catálogo entero en una tarde.

El código actual (malo)

const { data: connection } = await supabase
  .from('platform_connections')
  .select('*')
  .eq('id', connectionId)        // ← solo busca por ID
  .eq('is_active', true)         // ← y que esté activa
  .single();
// ¡Nunca pregunta si el user.id es el dueño!

El fix (bueno)

const { data: connection } = await supabase
  .from('platform_connections')
  .select('*, clients!inner(user_id)')  // ← join con clients
  .eq('id', connectionId)
  .eq('is_active', true)
  .single();

// Y verifica que el dueño coincida
if (connection.clients.user_id !== user.id && !isSuperAdmin) {
  return c.json({ error: 'Forbidden' }, 403);
}

S4 · Aceptamos JWT falsos sin verificar la firma

CRÍTICO

3 archivos: sync-shopify-metrics, fetch-shopify-products, fetch-shopify-analytics

¿Qué pasa?

Cuando alguien envía un token de autenticación (JWT), nosotros solo leemos lo que dice por dentro sin verificar que sea legítimo. Es como aceptar un cheque sin mirar la firma.

Imagina que...

Llega un tipo al banco con un cheque. El cajero no mira la firma, no llama al banco emisor, no nada. Solo lee el monto y entrega la plata. Si el cheque dice "$5.000.000", paga $5.000.000. ¿Lo escribió el dueño? ¿Lo escribió el ladrón? Nadie sabe.

Ejemplo concreto

Un atacante toma un JWT viejo y expirado que encontró en un log. Le cambia la fecha de expiración a mano. Lo manda a nuestro endpoint. Nosotros no verificamos la firma, así que pensamos que es legítimo y le damos acceso a productos, ventas, todo.

El código actual (malo)

// Decodifica el JWT manualmente, sin verificar firma
const payload = JSON.parse(atob(token.split('.')[1]));
const userId = payload.sub;
// ↑ Esto acepta CUALQUIER token, hasta uno hecho a mano

El fix (bueno)

// Usa la función oficial de Supabase que verifica la firma
const { data: { user }, error } = await supabase.auth.getUser(token);
if (error || !user) {
  return new Response('Unauthorized', { status: 401 });
}
const userId = user.id;

S5 · Fallback peligroso a una contraseña vieja

CRÍTICO

supabase/functions/shopify-fulfillment-webhooks/index.ts:286-290

¿Qué pasa?

Para verificar webhooks de Shopify usamos SHOPIFY_WEBHOOK_SECRET. Si esa variable falta, el código silenciosamente cae a usar SHOPIFY_CLIENT_SECRET (otra cosa distinta). Shopify dejó de soportar este patrón en 2022.

Imagina que...

Tienes una caja fuerte con una clave. Si alguien borra esa clave, la caja automáticamente empieza a usar tu fecha de cumpleaños como clave alternativa. Nadie te avisa. Tú piensas que está segura. Cualquiera que sepa tu cumpleaños la abre.

Ejemplo concreto

Algún día rotamos SHOPIFY_CLIENT_SECRET pensando que solo afecta el OAuth de login. Resultado: todos los webhooks de fulfillment empiezan a fallar silenciosamente. Los pedidos despachados ya no llegan al sistema. Felipe (W2) crea anuncios con stock incorrecto. Te enteras 2 semanas después.

Imagina que...

Tienes un buzón en tu casa. Cuando llega una carta importante, la metes en el buzón pensando "el cartero la va a recoger y procesar". Pero el buzón no tiene fondo: las cartas caen directo a la calle y se las lleva el viento.

async function enqueueBackgroundJob(payload: any) {
  console.log('[fulfillment] enqueued:', payload.id);
  // ← y ya. Nada más. No hay cola, no hay nada.
}

ALTOS 7 problemas serios

Altos — No son agujeros, pero te van a doler

Bugs y deuda técnica que tarde o temprano te muerden el tobillo.

A1 · Tenemos el código de Shopify duplicado en 2 lugares

ALTO

¿Qué pasa?

Casi todo el módulo Shopify (productos, analytics, collections, sync) está implementado dos veces: una en el backend de Cloud Run y otra en Edge Functions de Supabase. Las dos versiones son distintas. El frontend a veces llama una, a veces la otra.

Imagina que...

Tienes dos cocineros en tu restaurante. Los dos hacen "el mismo" risotto, pero uno usa parmesano y el otro grana padano. Uno tarda 20 min y el otro 30. El mesero al azar pide a uno o al otro. Cada cliente recibe una experiencia diferente. Y si quieres cambiar la receta, tienes que avisarle a los dos cocineros (y vas a olvidar a uno).

Ejemplo concreto

El analytics del backend tiene Lifetime Value, cohortes y funnel completo. El analytics de la edge function NO los tiene. Si un cliente abre el dashboard y por alguna razón se invoca la edge function, ve métricas pobres. Si invoca el backend, ve métricas ricas. Es ruleta.

El fix

Borrar las edge functions de fetch/sync y dejar SOLO las que tienen sentido como edge function: OAuth callback, webhooks públicos y GDPR.

A2 · Hay un flujo OAuth viejo que nadie limpió

ALTO

src/pages/OAuthShopifyCallback.tsx:97-152

¿Qué pasa?

El callback de OAuth tiene dos rutas internas: una nueva (que funciona) y una vieja (que ya no se usa pero quedó comentada y conectada). Algún día alguien va a tocar el archivo y "restaurar" el código viejo pensando que es el bueno.

Imagina que...

Renovaste tu casa. Tienes una llave nueva que abre la puerta nueva. Pero dejaste la cerradura vieja puesta en la puerta, "por si acaso". Cualquier ladrón que tenga la llave vieja todavía entra.

A3 · El panel de pedidos hace 2 llamadas en vez de 1

ALTO

src/components/client-portal/ShopifyOrdersPanel.tsx:97-101

¿Qué pasa?

Cuando el panel de pedidos carga, hace una llamada a Shopify. Si esa llamada no devuelve pedidos, automáticamente hace una segunda llamada con un parámetro distinto. Para clientes nuevos sin pedidos, esto significa quemar el doble del rate limit cada vez que abren el tab.

Imagina que...

Pides una pizza por teléfono. La pizzería te dice "no tenemos masa". En vez de colgar, les vuelves a llamar inmediatamente a preguntar lo mismo. Te dicen otra vez "no tenemos masa". Repites cada vez que abres el menú. La pizzería te bloquea.

A4 · Le mentimos a los clientes con datos inventados

CRÍTICO DE NEGOCIO

src/components/client-portal/metrics/ShopifyFunnelPanel.tsx:32-33

¿Qué pasa?

El panel de funnel muestra "sesiones, carritos abandonados, checkouts, compras". Si Shopify no nos da las sesiones (porque el cliente no tiene Shopify Plus), las inventamos multiplicando carritos × 5. Le mostramos al cliente un número que no existe.

Imagina que...

Vas al doctor. Te toma la presión pero el aparato está roto. En vez de decirte "no puedo medirla", el doctor se inventa un número basado en cuántos años tienes. Te dice "120/80" sin tener idea. Tú confías en él y tomas decisiones de salud con eso.

Ejemplo concreto

Un cliente ve "5.000 sesiones este mes" y dice "qué bien, mi tienda está creciendo". En realidad nunca medimos sesiones. Cuando contrata más anuncios y las "sesiones" no suben proporcionalmente, piensa que Steve Ads no funciona y se va.

El código actual (malo)

// Si no hay sesiones, las inventa multiplicando
const sessions = funnel.sessions || addToCarts * 5;
const addToCarts = funnel.addToCarts || checkouts * 2.5;

El fix (bueno)

// Si no hay dato, mostrar "—" (no inventarlo)
const sessions = funnel.sessions ?? null;
const addToCarts = funnel.addToCarts ?? null;
// En el render:
{sessions !== null ? sessions.toLocaleString() : '—'}

Imagina que...

Tienes un GPS que normalmente lee las coordenadas del satélite. Pero tiene grabada a fuego la dirección de tu casa anterior "por si falla el satélite". Te mudas, un día falla el GPS, y termina llevándote a tu casa vieja.

A7 · Borrado GDPR muy lento (puede colgar)

ALTO

supabase/functions/shopify-gdpr-webhooks/index.ts:117-138

¿Qué pasa?

Cuando un cliente desinstala la app y Shopify nos pide borrar sus datos por GDPR, el código borra una tabla a la vez, esperando que termine cada una. Para clientes con muchos datos puede tomar más de 2 minutos y exceder el timeout.

Imagina que...

Tienes que vaciar 10 cajones de tu escritorio. En vez de pedirle a 10 amigos que cada uno vacíe un cajón al mismo tiempo, vacías uno tú, esperas a terminar, vacías otro, esperas, y así. Tardas 10 veces más.
