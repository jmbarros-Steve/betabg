Qué hice con Shopify hoy

Dejé el sistema preparado para que funcione HOY con Custom Apps y mañana con App Store, sin reescribir nada.

La idea en 30 segundos

Imagina un portero de un edificio.

Antes, el portero tenía una sola llave maestra que abría todos los departamentos. Problema: cuando el edificio tenga muchos dueños, cada uno tiene SU llave. El portero no puede seguir usando una maestra.

Hoy le enseñé al portero a mirar el número del departamento primero, y automáticamente decidir si usa la llave del dueño (guardada en un llavero) o la maestra del edificio (colgada en su oficina). El portero no elige — el sistema elige por él.

¿Por qué hacía falta esto?

Antes (roto para Custom Apps)

Cuando un cliente se conectaba con Custom App, sus credenciales quedaban guardadas en la base de datos — pero el código que verifica los webhooks seguía leyendo una variable global SHOPIFY_WEBHOOK_SECRET que NO existe para ese cliente. Resultado: todos los webhooks fallaban con HMAC mismatch.

Ahora (dual mode)

Cada conexión queda marcada como custom_app o app_store. Cuando llega un webhook, el código mira la marca y saca las credenciales del lugar correcto. Un solo código, dos modos.

Cambio #1 — Columna nueva en la base de datos

Imagina que...

A cada ficha de cliente en el archivador le pegué un sticker de color: sticker verde = Custom App, sticker azul = App Store. Cuando necesites algo del cliente, primero mirás el sticker y sabés a qué gaveta ir. 

ALTER TABLE platform_connections
ADD COLUMN connection_mode text
  NOT NULL DEFAULT 'custom_app'
  CHECK (connection_mode IN ('custom_app', 'app_store'));

Default = custom_app → todas las conexiones existentes quedan marcadas automáticamente así. No hay que tocar nada manualmente.

Cambio #2 — El "portero" inteligente (Credential Resolver)

Imagina que...

Antes, cada función del backend leía las credenciales directo de las variables de entorno. Como si cada empleado de la empresa tuviera su propia copia de las llaves y las chequeara cada uno por su lado — un desastre cuando hay dos tipos de llaves.

Ahora TODOS le preguntan a una sola función: "¿cuáles son las credenciales de la tienda tienda-x.myshopify.com?" y esa función se encarga de mirar la base de datos, ver el sticker, y devolver las credenciales correctas.

Ejemplo concreto:

// ANTES (en cada archivo, repetido 10 veces)
const secret = process.env.SHOPIFY_CLIENT_SECRET;
if (!secret) return error;

// AHORA (una sola llamada en todos lados)
const creds = await resolveShopifyCredentials(shopDomain);
if (!creds) return error;
const secret = creds.webhookSecret; // ← el correcto, siempre

Cambio #3 — Webhooks que ya no fallan

Imagina que...

Shopify te llama al teléfono cada vez que hay una venta. Para saber que es realmente Shopify (no un estafador), usa un sello secreto que solo vos y ellos conocen. Antes, tenías un solo sello global. Ahora cada cliente tiene SU sello, y el sistema automáticamente saca el sello correcto del cajón de cada cliente antes de verificar.

Cambio #5 — Wizard mejorado

Validación de formato

Antes podías pegar cualquier garabato como Client Secret. Ahora se valida el formato (32+ hex o empieza con shpss_) ANTES de enviarlo al backend. Error visible al instante.

Timeout del polling

El wizard "esperaba para siempre" a que el usuario terminara de instalar. Ahora tiene un timeout de 10 minutos con botón "Reintentar detección".

URL hardcodeada eliminada

Antes tenía fallback a una URL hardcodeada de Cloud Run. Ahora solo usa VITE_API_URL. Si falta, avisa en consola.

Riesgos reales (no te voy a mentir)

⚠️ Riesgo 1 — Clientes Custom App existentes pueden no tener client_secret_encrypted

Si hay tiendas conectadas ANTES del wizard de Custom App (en el modelo viejo con SHOPIFY_CLIENT_SECRET global), esas conexiones NO tienen shopify_client_secret_encrypted en la DB. El resolver las marca como custom_app (default), busca el secret en DB, no lo encuentra → retorna null → webhook falla.

Mitigador ya aplicado: los 4 handlers de webhooks tienen fallback a env vars cuando el resolver retorna null. O sea, si SHOPIFY_WEBHOOK_SECRET sigue seteado globalmente, las conexiones viejas siguen funcionando.

Acción recomendada: antes de deploy, correr este query para ver si hay tiendas en esa situación: 

SELECT id, shop_domain, connection_mode,
  shopify_client_id IS NULL AS missing_client_id,
  shopify_client_secret_encrypted IS NULL AS missing_secret
FROM platform_connections
WHERE platform = 'shopify' AND is_active = true;
⚠️ Riesgo 2 — Migration falla si ya existe la columna

La migration usa ADD COLUMN IF NOT EXISTS → seguro contra rerun. Pero si CHECK (...) choca con data existente, falla. Como el default es custom_app y la columna es nueva, no debería pasar. Aun así, revisar logs al ejecutarla.

⚠️ Riesgo 3 — Doble función con el mismo nombre en shopify-oauth-callback.ts

Ese archivo YA tenía una función local resolveShopifyCredentials distinta. Yo NO la toqué porque funciona. Pero hay dos funciones con el mismo nombre en el proyecto (una en el lib, una en el archivo del callback). TypeScript lo permite porque están en scopes distintos, pero es confuso. Merece un rename en el futuro.

⚠️ Riesgo 4 — Edge functions y Cloud Run tienen handlers duplicados

Existen DOS implementaciones de cada webhook: una en supabase/functions/ (Deno) y otra en cloud-run-api/src/routes/shopify/ (Node). Ambas fueron arregladas. Pero solo UNA es la que realmente recibe los webhooks (la que se registra en registerWebhooks del OAuth callback → apunta a Cloud Run).

Las edge functions son zombies. Arreglé las dos por si se activa alguna eventualmente, pero se podrían borrar en un cleanup futuro (~600 líneas de código muerto).

✅ Lo que NO es riesgo
  • • TypeScript compila limpio (backend + frontend, exit 0)
  • • El frontend sigue funcionando igual que antes (no cambié el flujo visible del wizard)
  • • Los clientes nuevos del wizard Custom App quedan marcados correctamente
  • • El SHOPIFY_MODE=custom default hace que nada cambie hasta que decidas cambiarlo
